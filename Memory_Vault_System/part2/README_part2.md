# Part 2: Vault Control Program (Y86 Assembly)

The second part of the Memory Vault System exercises your understanding of control flow and memory operations in a simplified assembly language.  You will write a short program in **Y86 assembly**—a pedagogical subset of the x86‑64 instruction set—using a restricted set of instructions.  Your program will process an array of integers, compute an unlock code, and compare it against a target value provided by the system.  Based on the comparison, the vault will either open or remain locked.

## Instruction Set

This assignment uses a streamlined Y86 instruction set together with two small extensions to support array traversal and parity testing.  Refer to your course materials for detailed semantics.

| Instruction      | Description                                       |
|------------------|---------------------------------------------------|
| `irmovq $imm, %r`| Move immediate value `imm` into register `%r`.      |
| `irmovq label, %r`| Move the address label (e.g., `array`) into `%r`.   |
| `rrmovq %rA, %rB`| Copy contents of `%rA` into `%rB`.                  |
| `addq %rA, %rB`  | Add `%rA` to `%rB` and store the result in `%rB`.   |
| `subq %rA, %rB`  | Subtract `%rA` from `%rB` and store in `%rB`.       |
| `andq %rA, %rB`  | Bitwise AND `%rA` with `%rB`; store in `%rB`.  Sets ZF when the result is zero. |
| `mrmovq D(%rA), %rB` | Load memory at address `label` stored in `%rA` plus constant displacement `D` into `%rB`. |
| `mrmovq D(%rA,%rB), %rC` | Load memory at address `label` stored in `%rA` plus displacement `D` and a **dynamic offset** (in bytes) from `%rB` into `%rC`.  The dynamic offset allows pointer arithmetic.  Omitting `D` (i.e. `mrmovq (%rA,%rB), %rC`) implies a constant of zero. |
| `jmp label`      | Unconditional jump to `label`.                    |
| `je label`       | Jump to `label` if the zero flag is set.          |
| `jne label`      | Jump to `label` if the zero flag is not set.      |
| `halt`           | Terminate program execution.                      |

Your program **may not use any other instructions**; if you are accustomed to x86 assembly, restrict yourself to the subset above.  The zero flag (ZF) is set by arithmetic instructions (`addq`, `subq`, `andq`) when the result is zero.  Conditional jumps test this flag.

## Programming Task

You must complete the file `program_template.y86`.  The template sets up three registers before the `start` label:

* `%r1` holds the base address of an integer array.
* `%r2` contains the length of that array.
* `%r3` contains the target code against which you will compare.  When you run the simulator with the correct Part 1 password, `%r3` is set to the true target (equal to the result you should compute).  Without the correct password, `%r3` is offset so that your program’s result will not match.

Your task is to write a loop that iterates over the array, sums the values at even indices and subtracts the values at odd indices, and stores the result in `%r4`.  After the loop, **compare** your computed result (in `%r4`) to the target code (in `%r3`) without destroying `%r4`.  A common pattern is to copy `%r4` to a temporary register (e.g., `%r9`), subtract `%r3` from the copy, and then use `je` and `jne` based on the zero flag to branch to the labels `open` and `locked` respectively.  Both labels immediately call `halt` to terminate execution.

The provided Python interpreter (`y86_sim.py`) initialises memory so that labels such as `array`, `len`, and `target` map to their respective values.  It then executes your assembly program and reports the outcome.  At the end of execution, the simulator prints the final computed code, the target code, and whether the vault opens based on your program’s control flow.

## Running Your Program

To test your assembly code, run the simulator from within the `part2` directory.  The simulator now accepts up to two additional arguments: your student identifier and your recovered password from Part 1.  The general syntax is:

```
python y86_sim.py program_template.y86 <STUDENT_ID> [PASSWORD]
```

* `<STUDENT_ID>` (required) is your university ID (e.g., `U0000015860`).  It determines the seed used to personalise the array and vault.
* `[PASSWORD]` (optional) is the password you recovered in Part 1.  If you provide the correct password, the simulator will set the target code equal to the computed result.  Without the correct password, the target code is offset so that even a correct assembly program cannot open the vault.

Example:

```
python y86_sim.py program_template.y86 U0000015860 QMKRTA
```

If the password matches the one generated by Part 1 for the given student ID, the simulator will display a summary similar to the following:

```
--- Vault Control System ---

Final Code: 15
Target Code: 15

VAULT OPEN
Congrats, you unlocked 20% of your grade.
```

If you omit the password or enter an incorrect one, the target code will not match the computed result and the simulator will report **VAULT LOCKED**.

## Tips

* Use `%r5` as a loop counter and `%r6` as a byte‑offset register.  Initialise both to zero.  Each iteration, increment `%r5` by one and increment `%r6` by eight to advance to the next array element.
* The dynamic addressing form `mrmovq (%r1,%r6), %rX` loads the element at offset `%r6` from the base label stored in `%r1` (which the simulator initialises to the start of the array).  This allows you to traverse the array without manually computing constant offsets in the instruction stream.
* To distinguish even and odd indices, you can maintain a **sign register** that alternates between `1` and `-1`.  Alternatively, you can use the `andq` instruction with a mask of `1` to test the least significant bit of the loop counter.  A zero result means the index is even; a non‑zero result means it is odd.
* At the end of each iteration, update your accumulator `%r4` by adding or subtracting the loaded value based on its parity, then increment `%r6` and `%r5` to move to the next element.  When `%r5` equals the length `%r2`, exit the loop.
